# zod+nest (mock ts-rest project)

Небольшой моковый проект, демонстрирующий **contract-first** подход с **ts-rest** и **Zod**:
- контракт описан в общем месте (`shared`) и содержит **схемы** (Zod) + метаданные маршрутов;
- сервер (NestJS) реализует маршрут и старается вернуть ответ **строго по контракту**;
- клиент (React/Vite) использует тот же контракт для типобезопасного обращения к API.

## Концепт

### Contract-first (один контракт для клиента и сервера)
Контракт описывает:
- `method`, `path`
- `responses`: статус-код → схема тела ответа (Zod)

По идее это уменьшает рассинхрон “клиент ждёт одно — сервер отдаёт другое”.

### Runtime validation
Для обеспечения строгого соответствия:
- в контракте используется `z.object(...).strict()` — неизвестные поля должны считаться ошибкой на уровне валидации;
- на сервере включается `validateResponses: true` в `@TsRestHandler`;
- дополнительно в примере показана явная проверка через `Pokemon.safeParse(...)`, чтобы корректно сформировать ответ `400` с понятным телом ошибки.

### shared как локальный пакет без публикации
`shared` оформлен как локальный npm-пакет (например, через зависимость `file:../shared` в `server` и `client`), чтобы:
- контракт импортировался по имени (`@repo/shared`);
- в репозитории не нужно публиковать пакет в npm registry.

## Стек

- Backend: **NestJS**, **@ts-rest/nest**, **Zod**
- Frontend: **React**, **@ts-rest/core** (typed client)
- Shared: **контракт + Zod схемы**

## Пример домена: Pokemon (mock)

Маршрут:
- `GET /pokemon/:id`

В текущем моковом коде сервис возвращает “заглушечные” данные Pokemon, а контракт задаёт структуру ответа.
Если ответ не соответствует схеме (например, лишние/отсутствующие поля), сервер возвращает ошибку `400`.

## Структура (упрощённо)

- `shared/`
  - `contract.ts` — Zod схемы и `pokemonContract`
  - `package.json` — метаданные локального пакета (без публикации)
- `server/`
  - `src/pokemon/pokemon.controller.ts` — подключение контракта и обработка ответа
  - `src/pokemon/pokemon.service.ts` — мок-данные
- `client/`
  - `src/App.tsx` — вызов API через `initClient(pokemonContract, ...)`

## Как это запустить (в общих чертах)

1) Установить зависимости:
- `server/`: `npm install`
- `client/`: `npm install`

2) Запустить:
- `server`: `npm run start:dev`
- `client`: `npm run dev`

Ожидается, что фронт будет ходить на backend (например `http://localhost:3000`), а backend должен иметь CORS для `localhost:5173`.

## Почему это полезно (что именно демонстрирует мок)

- что контракт действительно может быть “источником правды” для формы ответа;
- как включать проверку соответствия контракту на сервере;
- как возвращать `400` в виде, согласованном с `responses` в контракте;
- как клиент становится типобезопасным, используя контракт из `shared`.

## Дальнейшие улучшения (идеи)

- вынести ручную `safeParse` логику в отдельный helper/мидлварь, чтобы контроллеры были короче;

- сделать контракт валидации без ручного `safeParse` (если настроить в ts-rest/nest именно нужный режим проверки);
- добавить генерацию OpenAPI (если планируешь документацию).
